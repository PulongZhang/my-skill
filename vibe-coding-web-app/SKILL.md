# Vibe Coding Web App

## 核心目标

使用一套简单、成熟、适合 AI 辅助开发的固定技术栈，规划和开发个人项目、内部管理系统及中小型 Web 应用。默认从模块化单体开始，以明确需求、类型边界、测试、文档和小步验证约束代码生成，不提前引入微服务或复杂基础设施。

详细技术栈、目录结构、编码规范和部署示例见 [`references/framework.md`](references/framework.md)。只在任务涉及对应细节时读取该参考文件，不要为了简单问答加载全部内容。

## 先判断任务类型

将任务归入以下一种或多种模式：

1. **项目规划**：梳理需求、数据模型、接口、页面、权限、架构和实施阶段。
2. **项目初始化**：创建 Vue 3 + FastAPI + PostgreSQL 的单仓库基础工程。
3. **功能实现**：在符合该技术栈的项目中完成一个可验证的前后端业务功能。
4. **架构检查**：检查项目是否偏离框架原则、存在不必要设施或缺少关键边界。
5. **部署配置**：配置 Docker Compose、OpenResty、Cloudflare 或生产部署说明。

若用户只要求其中一个模式，不要扩展到其他模式。

## 新项目与已有项目

### 新项目

使用本框架作为默认方案，但先确认无法从请求中得出的必要信息：

- 项目目标和首个业务功能；
- 核心数据字段与业务规则；
- 页面、角色和权限；
- 目标目录；
- 需要规划文档还是直接创建工程；
- 部署域名或环境要求（仅当任务涉及部署）。

已有明确答案时直接执行，不重复提问。非必要选项采用框架默认值，不把技术选型问题全部抛给用户。

### 已有项目

先读取仓库现状，包括：

- 项目说明和仓库级指令；
- 目录结构；
- `package.json`、锁文件和前端配置；
- `pyproject.toml`、锁文件和后端配置；
- 数据库、迁移、测试和部署配置；
- 已有代码的命名、分层和风格。

优先遵循已有项目约定。若项目使用 React、Django、Java、Go、其他数据库或其他部署方式，不要仅因本框架的默认选型而替换它们。只有用户明确要求迁移时，才先给出迁移范围、影响和验证方案，再实施迁移。

## 默认技术方案

仅在新项目或用户明确要求采用本框架时使用：

- 前端：Vue 3、TypeScript、Vite、Vue Router、Pinia、Axios、Element Plus、Tailwind CSS、pnpm；
- 后端：Python、FastAPI、Pydantic、SQLAlchemy 2、Alembic、PostgreSQL、psycopg 3、uv、Uvicorn、Ruff、Pytest；
- 部署：Docker、Docker Compose、1Panel、OpenResty、Cloudflare；
- 版本管理：单个 Git 仓库，使用简单主干开发和 Conventional Commits；
- 架构：模块化单体，前后端共用域名，API 使用 `/api/v1` 前缀。

Cloudflare 和 OpenResty 只承担反向代理职责。不要默认引入 Cloudflare Pages、Workers、R2 或 OpenResty Lua 业务逻辑。

## 需求规格

开始实现业务功能前，至少明确：

- 目标；
- 数据字段；
- 业务规则；
- API；
- 页面；
- 权限；
- 验收条件。

如果缺失信息会改变数据结构、权限边界或用户可见行为，应先询问用户。如果可以从现有代码、文档或惯例中可靠确定，则自行读取并采用，不要提无必要的问题。

## 执行原则

### 保持最小范围

- 每次只完成用户要求的一个可验证目标；
- 不添加未要求的通用框架、扩展点和配置项；
- 不为每张表机械创建 Repository，简单业务允许 Service 直接使用 SQLAlchemy；
- 不顺手重构无关代码；
- 只清理由本次修改造成的无用代码和配置。

### 避免过度设计

没有明确业务需求时，不引入：

- Kubernetes；
- 微服务；
- Kafka；
- Elasticsearch；
- GraphQL；
- CQRS；
- 事件溯源；
- 自研权限引擎；
- 多数据库兼容层。

Redis、异步任务、WebSocket 等组件只在需求明确需要时加入。

### 保护敏感信息

- 提交 `.env.example`，但不要读取、展示、写入或提交真实密钥；
- 不把数据库密码、JWT 密钥、Cloudflare Token、服务器私钥或第三方 API Key 写入代码和文档；
- 示例配置使用占位符；
- 项目名、目录、数据库名、域名和部署目标来自用户输入或项目现状，不得写死参考文档中的示例值。

## 各模式工作流

### 项目规划

1. 根据输入整理最小需求规格。
2. 列出必要的数据实体、关键字段和约束。
3. 定义 `/api/v1` 下的接口边界。
4. 定义页面、操作入口、角色和权限。
5. 给出模块化单体结构和必要依赖，不加入推测性设施。
6. 将实施过程拆成小阶段，每阶段写明可观察的验收条件。

除非用户要求创建文件，否则只输出规划，不初始化工程。

### 项目初始化

1. 检查目标目录，避免覆盖已有项目或未说明的文件。
2. 创建单仓库目录和最小文档结构。
3. 初始化 Vue 3 前端，验证开发入口和生产构建。
4. 初始化 FastAPI 后端，提供 `GET /api/v1/health`。
5. 接入 PostgreSQL、SQLAlchemy 和 Alembic，验证连接与迁移。
6. 添加 Docker Compose，验证 Web、API 和 PostgreSQL 服务。
7. 仅在用户要求部署时添加 OpenResty、Cloudflare 或 1Panel 配置。
8. 创建 `.env.example` 和适合实际工程的 `.gitignore`。

不要一次生成大量占位业务代码。先让最小工程可运行，再实现首个业务功能。

### 功能实现

通常按照以下顺序执行，但应服从现有项目结构和任务范围：

1. 明确验收条件；
2. 设计或调整数据模型与迁移；
3. 实现后端 Schema、Service 和 Router；
4. 添加能复现需求和边界条件的后端测试；
5. 基于 OpenAPI 同步或生成前端类型；
6. 实现 API 调用、状态管理和页面；
7. 执行端到端或最接近真实使用路径的验证。

若用户只要求后端、前端、数据库或文档，不要擅自实现完整纵向功能。

### 架构检查

检查并按严重程度输出：

- 与用户目标或框架核心原则冲突的问题；
- 前后端类型或接口边界不一致；
- 路由、Service、Repository 职责明显混乱；
- 开发与生产数据库或运行方式不一致；
- Web、API 或数据库端口不必要地暴露公网；
- 密钥、环境文件或生产数据可能被提交；
- 没有明确需求却引入复杂基础设施；
- 缺少可执行的验证方式。

不要把个人风格偏好列为必须修改项。对已有项目，仅报告与其实际目标相关的问题，不要求为了匹配参考目录而重排所有文件。

### 部署配置

1. 先确认实际域名、端口、证书终止位置和部署目录。
2. Web 与 API 宿主机端口默认只绑定 `127.0.0.1`。
3. PostgreSQL 默认不映射宿主机公网端口。
4. OpenResty 保留 `/api/v1` 路径，不默认重写 API 前缀。
5. Cloudflare 使用严格的源站 HTTPS 校验，不使用 Flexible SSL。
6. 输出或修改配置后，说明实际执行过的配置检查和仍需在服务器完成的人工步骤。

## 验证

先读取项目配置，再决定运行哪些命令。只运行项目实际存在或本次任务明确添加的检查：

- 前端：以 `package.json` 的 scripts 为准，常见检查包括 lint、typecheck、test、build；
- 后端：以 `pyproject.toml` 为准，常见检查包括 Ruff、Pytest 和 Alembic；
- 部署：检查 Compose 配置，并在适合当前环境时启动服务验证健康状态；
- 功能：优先验证用户实际操作路径，而不只依赖静态检查。

不要声称运行过不存在、被跳过或失败的检查。若检查失败，报告命令、关键错误和当前状态，不要掩盖失败。

## Git 边界

- 将工作组织成适合独立提交的小阶段，并可给出 Conventional Commit 建议；
- 不在用户未明确要求时执行 `git commit`、`git push`、创建远程仓库或创建 PR/MR；
- 不在默认分支上主动开展大范围开发；
- 执行 Git 写操作前检查当前状态，保留用户已有的未提交改动；
- 不修改、删除或提交与当前任务无关的文件。

## 结果报告

完成实施任务后，简要报告：

1. 完成了什么；
2. 修改或创建了哪些关键文件；
3. 执行了哪些验证及结果；
4. 哪些检查未执行以及原因；
5. 仍需用户完成的部署、凭据或外部服务步骤。

规划或架构检查任务则直接给出方案或发现，不套用实施报告格式。
