# 标准会议纪要 DOCX

## 核心目标

把钉钉会议 AI 听记导出的纪要 Markdown、文字记录 Markdown、会议转写、AI 摘要、问题记录或历史纪要整理为可直接提交的标准中文会议纪要 DOCX。

最终产物必须以 `references/标准会议纪要模板.docx` 为固定版式模板，保持其配色、表格、页眉、字体、边框、合并单元格、页面设置和整体布局一致。该模板由历史纪要清除真实会议数据后生成；不要再以包含真实会议内容的历史文件作为直接输出模板，也不要从空白文档重新搭建版式。

## 输入材料

优先支持钉钉会议 AI 听记导出的两类 Markdown 文档：

- 纪要文档：通常以 `# 纪要` 开头，包含主题、时间、参与人、会议背景、分章节摘要、待办或总结。
- 文字记录文档：通常包含创建时间、转写时长、关键词和 `## **转文字**`，正文按“发言人 + 时间戳 + 发言内容”排列。

处理钉钉听记导出文件时，先读取 `references/dingtalk-input-templates.md`，按其中模板抽取关键信息。也可以使用会议转写稿、AI 摘要、问题记录、历史纪要和其他 DOCX 作为内容来源，但最终版式仍以本 skill 的固定 DOCX 模板为准。

## 工作流程

1. 收集输入材料：优先读取用户给出的钉钉会议 AI 听记纪要 Markdown、文字记录 Markdown；再读取会议转写稿、AI 摘要、问题记录和历史纪要。
2. 抽取关键信息：若输入为钉钉听记导出文件，先按 `references/dingtalk-input-templates.md` 整理纪要文档和文字记录文档的关键信息。
3. 提取基础信息：识别会议主题、时间、形式、参会人员、主持人、会议类型、会议结论、记录人和日期。
4. 梳理会议内容：只保留明确讨论过的事实、结论、问题、责任人、解决方案和遗留事项。
5. 读取 `references/standard-format.md`，按其中字段和内容规则形成结构化纪要内容。
6. 将结构化纪要数据写入 UTF-8 JSON，调用 `script/generate_meeting_minutes_docx.py` 复制 `references/标准会议纪要模板.docx` 并生成输出文件；不要修改模板原文件。
7. 保留模板格式：不得删除或重建页眉、主表、嵌套表格、绿色字段底色、边框、合并关系、字体和页面设置；新增表格行应复制模板数据行后再替换内容。
8. 设置会议类型：保留模板中的 Wingdings 2 复选框，只把唯一匹配项设为选中，其他项保持未选中。
9. 命名 DOCX 文件：严格使用“会议主题会议纪要YYYYMMDD.docx”。
10. 自检输出：打开或解析生成的 DOCX，核对文件名、页眉、配色、表格结构、会议类型勾选、问题记录、遗留问题、会议总结、记录人和日期。

## DOCX 生成约束

- 默认生成 `.docx`，因为固定模板是 DOCX，能完整保留样式和结构。
- 用户明确要求旧版 `.doc` 时，先生成并验证 DOCX，再使用可用的 Word 或 LibreOffice 转换；不要直接构造二进制 `.doc`。
- 直接复制模板后修改内容，不要新建空白 `Document()` 后重画表格。
- 不得修改页眉“公司内部会议”及其中的图片、关系文件或定位。
- 保持模板字段标题单元格的绿色填充 `A8D08D`。
- 保持主表 4 列布局以及“会议记录”“遗留问题”“会议总结”区域的合并单元格结构。
- 问题记录表字段固定为：序号、提问者、问题内容、跟进者、解决方案、结果。
- 遗留问题表字段固定为：序号、问题描述、跟进者、预期完成时间。
- 问题或遗留事项数量变化时，复制现有数据行的完整 XML/格式后填值；不要只追加默认样式行。
- 内容替换应尽量复用原段落和 run，避免用 `cell.text = ...` 覆盖复杂单元格造成字体、段落和嵌套表格样式丢失。
- 记录人与日期必须位于同一段落、同一行：左侧为“记录人：”，右侧为“日期：”；使用右对齐制表位，不使用大量空格模拟对齐。
- 除非用户明确要求，不额外输出 Markdown、HTML、PDF 或纯文本版本。

## 生成脚本

使用 `script/generate_meeting_minutes_docx.py` 生成最终 DOCX：

```bash
python script/generate_meeting_minutes_docx.py \
  --input meeting-data.json \
  --output 会议主题会议纪要YYYYMMDD.docx
```

脚本输入 JSON 使用以下字段：

- `meeting_topic`、`project_number`、`meeting_date`、`meeting_format`
- `participants`、`host`、`meeting_type`、`conclusion`
- `purpose`、`issues`、`remaining_issues`、`summary`、`recorder`

脚本负责复制模板、填充字段、勾选唯一会议类型、按数量复制问题表格行、将记录人与日期保持在同一行，并在保存后恢复模板的页眉和样式部件。

## 信息缺失处理

- 缺少项目编号时填写空值，不要编造。
- 缺少会议形式时根据材料判断；无法判断则填写空值。
- 缺少会议结论时根据材料写“待确认”；不要默认写“通过”。
- 缺少记录人时优先用主持人；仍无法判断则填写空值。
- 没有遗留问题时在遗留问题表中写“无”。
- 责任人、预期完成时间不明确时写“待确认”。

## 写作口径

- 使用正式、客观、简洁的中文。
- 不写 AI 洞察、章节时间线、录音摘要封面、图片链接或非正式待办清单。
- 不夸大会议价值，不写“显著提升”“有效保障”等评价性表述。
- 不保留转写中的口头禅、重复语气词、识别错误和无关寒暄。
- 问题记录要按“谁提出、什么问题、谁跟进、怎么处理、结果状态”整理。
- 解决方案必须来自会议材料；无法确认时写“待确认”或“后续跟进”，不要自行补方案。

## 文件命名

生成 `.docx` 文件时，文件名必须使用：`会议主题会议纪要YYYYMMDD.docx`。

规则：

- `会议主题` 使用纪要中的正式会议主题，去掉空格、冒号、斜杠、引号等不适合作为文件名的符号。
- `YYYYMMDD` 使用会议时间，不使用生成文件的当天日期。
- 不要添加“纪要_”“标准版”“整理版”“final”等额外前后缀。
- 示例：会议主题为“流程引擎整体设计与功能模块评审”，会议时间为 `2026-02-05`，文件名为 `流程引擎整体设计与功能模块评审会议纪要20260205.docx`。

## 自检清单

输出前逐项检查：

- 最终产物是 DOCX；除非用户明确要求，不以 Markdown 作为最终产物。
- DOCX 文件名符合 `会议主题会议纪要YYYYMMDD.docx`。
- 页眉与模板一致，显示“公司内部会议”，页眉图片和位置未丢失。
- 主表、问题记录表、遗留问题表与模板的边框、列宽、合并关系和字体一致。
- 字段标题单元格保持模板绿色 `A8D08D`，未引入其他配色。
- 标题为“会议纪要”。
- 会议信息包含会议主题、项目编号、会议时间、会议形式、参会人员、主持人、会议类型、会议结论。
- 会议类型只勾选一个最匹配项；不确定时勾选“其他”。
- 问题记录表字段为：序号、提问者、问题内容、跟进者、解决方案、结果。
- 遗留问题表字段为：序号、问题描述、跟进者、预期完成时间。
- 末尾包含记录人和日期。
- 生成文件可被 Word/WPS 正常打开，且没有损坏提示。
