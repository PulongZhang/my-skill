# 标准会议纪要 Markdown

## 核心目标

把钉钉会议 AI 听记导出的纪要 Markdown、文字记录 Markdown、会议转写、AI 摘要、问题记录或历史纪要整理为可直接提交的标准中文会议纪要 Markdown，格式对齐公司 DOCX 会议纪要模板。

## 输入材料

优先支持钉钉会议 AI 听记导出的两类 Markdown 文档：

- 纪要文档：通常以 `# 纪要` 开头，包含主题、时间、参与人、会议背景、分章节摘要、待办或总结。
- 文字记录文档：通常包含创建时间、转写时长、关键词和 `## **转文字**`，正文按“发言人 + 时间戳 + 发言内容”排列。

处理钉钉听记导出文件时，先读取 `references/dingtalk-input-templates.md`，按其中模板抽取关键信息，再生成标准会议纪要。也可以使用会议转写稿、AI 摘要、问题记录、历史纪要和 DOCX 模板作为输入材料。

## 工作流程

1. 收集输入材料：优先读取用户给出的钉钉会议 AI 听记纪要 Markdown、文字记录 Markdown；再读取会议转写稿、AI 摘要、问题记录、历史纪要和 DOCX 模板。
2. 抽取关键信息：若输入为钉钉听记导出文件，先按 `references/dingtalk-input-templates.md` 中的模板整理纪要文档和文字记录文档的关键信息。
3. 提取基础信息：识别会议主题、时间、形式、参会人员、主持人、会议类型、会议结论、记录人和日期。
4. 梳理会议内容：只保留明确讨论过的事实、结论、问题、责任人、解决方案和遗留事项。
5. 生成 Markdown：严格使用 `references/standard-format.md` 中的结构和表格字段。
6. 命名 Markdown 文件：严格使用“会议主题会议纪要YYYYMMDD.md”。
7. 自检输出：核对文件名、会议类型勾选、问题记录表、遗留问题、会议总结、记录人日期是否完整。

## 信息缺失处理

- 缺少项目编号时填写空值，不要编造。
- 缺少会议形式时根据材料判断；无法判断则填写空值。
- 缺少会议结论时根据材料写“待确认”；不要默认写“通过”。
- 缺少记录人时优先用主持人；仍无法判断则填写空值。
- 没有遗留问题时在遗留问题表中写“无”。
- 责任人、预期完成时间不明确时写“待确认”。

## 写作口径

- 使用正式、客观、简洁的中文。
- 不写 AI 洞察、章节时间线、录音摘要封面、图片链接或非正式待办清单。
- 不夸大会议价值，不写“显著提升”“有效保障”等评价性表述。
- 不保留转写中的口头禅、重复语气词、识别错误和无关寒暄。
- 问题记录要按“谁提出、什么问题、谁跟进、怎么处理、结果状态”整理。
- 解决方案必须来自会议材料；无法确认时写“待确认”或“后续跟进”，不要自行补方案。

## 输出格式

生成最终纪要前，读取 `references/standard-format.md`，按其中的模板输出完整 Markdown。

## 文件命名

生成 `.md` 文件时，文件名必须使用：`会议主题会议纪要YYYYMMDD.md`。

规则：

- `会议主题` 使用纪要中的正式会议主题，去掉空格、冒号、斜杠、引号等不适合作为文件名的符号。
- `YYYYMMDD` 使用会议时间，不使用生成文件的当天日期。
- 不要添加“纪要_”“标准版”“整理版”“final”等额外前后缀。
- 示例：会议主题为“流程引擎整体设计与功能模块评审”，会议时间为 `2026-02-05`，文件名为 `流程引擎整体设计与功能模块评审会议纪要20260205.md`。

## 自检清单

输出前逐项检查：

- Markdown 文件名符合 `会议主题会议纪要YYYYMMDD.md`。
- 标题为 `# 会议纪要`。
- 会议信息表包含会议主题、项目编号、会议时间、会议形式、参会人员、主持人、会议类型、会议结论。
- 会议类型只勾选一个最匹配项；不确定时勾选“其他”。
- 问题记录表字段为：序号、提问者、问题内容、跟进者、解决方案、结果。
- 遗留问题表字段为：序号、问题描述、跟进者、预期完成时间。
- 末尾包含 `记录人：` 和 `日期：`。
