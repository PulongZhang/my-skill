# 代码评审评论格式规范

用 azure-devops-pr-code-review 技能做静态 code review、向 PR 发评论时遵循本规范，保证评论**可执行、可追溯、分级清晰**。

默认做**静态代码评审**：只读 PR diff 和必要上下文，不运行测试/构建/服务，不修改代码。只有用户明确要求“验证/跑测试/修复”时，才进入执行或修改流程；只有用户明确要求“发评论/作出评论/评论到 PR”时，才调用接口发布评论。

## 评审产出结构（总结必发，行内仅必改）

每次发布到 PR 的代码评审都要先发**一条总结评论**；行内评论只用于 `【必改】`，且必须钉到具体代码行：

1. **一条总结评论（普通评论，必发）**：覆盖整个 PR——一句话结论 + 按严重程度分级的编号清单（含 `【必改】`/`【建议】`/`【确认】`/`【提示】`）+ 已核对无问题项。模板见 §3。
2. **行内评论（仅 `【必改】`）**：每个必改具体问题一条，钉到对应文件的具体行，一条只讲一个问题。模板见 §2。

两者配合：总结评论里用 `<file:line>` 引用；`【必改】` 条目写"详见对应行内评论"，行内评论承载具体修复细节；`【建议】`/`【确认】`/`【提示】` 条目只写在总结评论里，不发布行内评论。

- **不要给 `【建议】`/`【确认】`/`【提示】` 写行内评论**：非阻塞事项避免打断作者逐行处理。
- **不要把 `【必改】` 只放总结不留行内**：作者无法定位到具体代码。
- **不要只发行内不发总结**：作者看不到整体结论与必改项全貌。
- 没有 `【必改】` 的 PR 只发总结评论（结论 = 可合并 / 待确认；"已核对无问题"列出审过的关键点）。

## 1. 严重程度标签（每条行首用 `【】` 标注）

| 标签 | 含义 | 阻塞合并 |
|---|---|---|
| `【必改】` | 正确性 bug / 安全 / 编译或测试失败 / 数据丢失 / i18n 缺失等 | 是 |
| `【建议】` | 设计与可维护性、较可能的隐患、与仓库约定不一致 | 否，应处理 |
| `【确认】` | 需作者确认意图，或验证某行为变更的影响 | 否，等回复 |
| `【提示】` | 风格、可选优化、DRY 等 nit | 否，不强求 |

分级务必诚实：拿不准的标 `【确认】` 而非 `【必改】`；主观偏好一律 `【提示】` 且不强求。

## 2. 行内评论（仅 `【必改】`，钉到具体行）

只给 `【必改】` 写行内评论；`【建议】`/`【确认】`/`【提示】` 不写行内评论。每条必须钉到具体代码行，只讲一个问题，结构 = **标签 + 一句话问题 → 影响 → 修复建议**：

```
【必改】<一句话问题>
<为什么是问题 / 影响范围>
建议：<具体修复，含字段名/方法名/示例>
```

发评论命令：

```bash
python scripts/azdo_client.py add-comment <PR> \
  --file "<path>" --line <N> --change-tracking-id <ID> --content "..."
```

- `changeTrackingId`：用 `pr-changes` 命令取（对应文件那一行的序号）。
- `--line`：用本地 `git diff origin/<目标分支>...origin/<源分支>` 里 `+` 行的行号。

## 3. 总结评论（普通评论，一条盖全 PR）

结论先行 → 分级编号列出 → 已核对项收尾：

```
【代码评审 · PR {id} · <一句话结论：可合并 / 需改后合并 / 不建议合并>】

■ 必改（N）
1. <file:line> — <问题> → <修复>（详见对应行内评论）
■ 建议（N）
2. <file:line> — <问题> → <修复>（仅总结记录，不发行内评论）
■ 待确认（N）
3. <file:line> — <需作者确认的意图/影响>（仅总结记录，不发行内评论）
■ 提示（N）
4. <file:line> — <可选优化/风格提示>（仅总结记录，不发行内评论）
■ 已核对无问题
- <本次特意核对、确认正确的关键点，如前置条件/并发/回滚，让作者知道审过>
```

## 4. 原则

- **可执行**：每条给 `file:line`、给原因/影响、给具体修复；不写"这里不太好"这类空泛话。
- **行内克制**：行内评论只承载 `【必改】`，且必须钉到具体代码行；`【建议】`/`【确认】`/`【提示】` 进入总结评论，不单独打到代码行。
- **分级诚实**：见 §1，主观偏好不冒充必改。
- **不刷屏**：同类小问题合并成一条；纯夸奖并入结论或"已核对无问题"，不单独发。
- **核对留痕**：审过且正确的关键路径写进"已核对无问题"，避免作者重复答疑。
- **中文撰写**，与本仓库及团队习惯一致。
- **先读代码再评**：取改动用本地 `git diff origin/<目标>...origin/<源分支>`，必要时 `git show` 读上下文，不凭 diff 片段猜测。
